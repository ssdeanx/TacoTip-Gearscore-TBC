## Entry 2 — 2026-07-11 — prism-full (tooltip mover trace)

### Pass 1 — Lifecycle Map

#### State Machine

```
NOT_CREATED → [TacoTip_CustomPosEnable(show)] → CREATED + HIDDEN
CREATED + HIDDEN → [TacoTip_CustomPosEnable(true)] → CREATED + VISIBLE
CREATED + VISIBLE → [OnShow on TacoTipDragButton] → SHOWN_ACTIVE
SHOWN_ACTIVE → [show=true param is re-evaluated] → HIDDEN or VISIBLE
CREATED/HIDDEN/VISIBLE → [TacoTipDragButton:Hide()] → HIDDEN
ANY STATE → [TacoTipDragButton:_Disable()] → DISABLED + custom_pos=nil
```

#### Creation (first call to TacoTip_CustomPosEnable)

| Property | Value |
|---|---|
| Type | Button (Clickable + Draggable) |
| Parent | UIParent |
| Strata | TOOLTIP |
| FrameLevel | 999 |
| Size | 32×32 |
| NormalTexture | `Interface\MINIMAP\TempleofKotmogu_ball_green` |
| Registered mouse | LeftButton drag, MiddleButtonUp, RightButtonUp |

Creation is idempotent — guarded by `if (not TacoTipDragButton)`. The second branch of `TacoTip_CustomPosEnable` always runs `_Enable()` then conditionally shows/hides the mover.

**Script handlers registered at creation:**

| Handler | Purpose |
|---|---|
| OnDragStart | Starts moving, installs OnUpdate to follow GameTooltip position during drag |
| OnDragStop | Stops moving, reads `GetPoint()` → writes `TacoTipConfig.custom_pos`, calls syncTooltipMoverPosition(true), calls refreshOptionsUI |
| OnClick (middle) | Rotates `TacoTipConfig.custom_anchor` through `TOPRIGHT→BOTTOMRIGHT→BOTTOMLEFT→CENTER→TOPLEFT→TOPRIGHT`, calls ShowExample, calls refreshOptionsUI |
| OnClick (right) | Opens StaticPopup with 3 buttons: SAVE / CANCEL / RESET (middle button = RESET via OnAlt) |
| OnShow | Starts 1-second `NewTicker` calling `ShowExample`, hooks GameTooltip OnShow/OnHide via Detours:ScriptHook |
| OnHide | Cancels ticker, unhooks GameTooltip OnShow/OnHide |

#### ShowExample
- Calls `GameTooltip_SetDefaultAnchor(GameTooltip, UIParent)` (pre-reset before the securefunc hook)
- Sets unit to "player" (triggers onTooltipSetUnit with full GS/talent data)
- Adds three instruction lines
- Shows the tooltip

#### Ticker loop (1s)
- Every second during Show: calls `ShowExample` → re-populates GameTooltip with player data

#### Registered hooks during mover Show:
- **GameTooltip:OnShow** → `onMoverGameTooltipShow`: if tooltip shows non-player unit, calls `ShowExample` (reverts to player)
- **GameTooltip:OnHide** → `onMoverGameTooltipHide`: if mover is shown, calls `ShowExample` (restores player tooltip)

#### Teardown sequence (_Disable)
1. Hides TacoTipDragButton (triggers OnHide → cancels ticker, unhooks GameTooltip)
2. Hides GameTooltip
3. Clears GameTooltip points
4. Clears `TacoTipConfig.custom_pos`
5. Optionally clears `TacoTipConfig.custom_anchor` (preserved when called from checkbox `false`)

#### Public entry points (how the mover gets shown)
| Trigger | Call chain |
|---|---|
| `/tacotip custom/unlock/move/mover` | → `showTooltipMover()` → `TacoTip_CustomPosEnable(true)` |
| "Mover" button in options root page | → `showTooltipMover()` → `TacoTip_CustomPosEnable(true)` |
| "Mover" button in Positioning page | → `showTooltipMover()` → `TacoTip_CustomPosEnable(true)` |
| Options "Custom Tooltip Position" checkbox ON | → `TacoTip_CustomPosEnable(false)` (loads mover but hides it) |

#### Default Position
`getDefaultTooltipMoverPosition()` → `{custom_anchor or "TOPLEFT", custom_anchor or "TOPLEFT", 0, 0}`
- Anchor point defaults to TOPLEFT
- X/Y offset defaults to 0,0
- Positioned relative to UIParent
- Result: the green dot sits at the top-left corner of the screen, offset 0,0

### Pass 2 — Data Flow & Synchronization

#### Config keys and their roles

| Key | Type | Written by | Read by | Persisted? |
|---|---|---|---|---|
| `custom_pos` | `{string, string, number, number}` | OnDragStop, _Disable, _ResetPosition, _Enable, checkbox OnClick, slash "default", resetCfg | `hooksecurefunc("GameTooltip_SetDefaultAnchor")`, `TacoTipDragButton:SetPoint`, `getDefaultTooltipMoverPosition`, `syncTooltipMoverPosition`, ShowExample, _Enable, _Save, _Disable, _ResetPosition | Yes (SavedVariables) |
| `custom_anchor` | `string` (TOPRIGHT/BOTTOMRIGHT/BOTTOMLEFT/CENTER/TOPLEFT) | Option dropdown, middle-click cycle, slash "anchor topleft" etc | `getDefaultTooltipMoverPosition`, `hooksecurefunc`, `syncTooltipMoverPosition`, option panel, _Disable(preserveAnchor) | Yes |
| `anchor_mouse` | `boolean` | Checkbox OnClick, _Enable | `hooksecurefunc`, checkboxes disabled state | Yes |
| `anchor_mouse_world` | `boolean` | Checkbox OnClick | `hooksecurefunc` | Yes |
| `anchor_mouse_spells` | `boolean` | Checkbox OnClick | `hooksecurefunc` | Yes |

#### Synchronization paths

**Config → Visual (mover position):**
1. `TacoTip_CustomPosEnable` → `_Enable` → `syncTooltipMoverPosition(false)` → `TacoTipDragButton:ClearAllPoints()` + `SetPoint(pos[1], UIParent, pos[2], pos[3], pos[4])`
2. After drag stop: `onDragButtonDragStop` → `TacoTipConfig.custom_pos = {from, to, x, y}` → `syncTooltipMoverPosition(true)` → same setpoint cycle + ShowExample

**Config → Visual (tooltip anchor):**
1. `hooksecurefunc("GameTooltip_SetDefaultAnchor")` fires every time Blizzard re-anchors
2. If `custom_pos` is set: `GameTooltip:SetOwner(TacoTipDragButton, "ANCHOR_NONE")` → `GameTooltip:SetPoint(custom_anchor, TacoTipDragButton, custom_anchor)`
3. This works because the mover button position was already synced

**Option panel ↔ Config sync:**
- `Refresh()` on positioning page: reads TacoTipConfig keys and updates checkbox checked states, dropdown values, button enabled state
- Checkbox OnClick handlers: write TacoTipConfig, then call `modernGetConfig()` which updates the root-page summary text
- Mover button `_Enable`/`_Disable`: directly manipulates the global-named option controls (`_G.TacoTipOptCheckBoxCustomPosition`, etc.)

#### Divergence paths found

1. **Gap: `TacoTipDragButton` position during reload.** `custom_pos` is persisted in SavedVariables, but `TacoTipDragButton` is re-created fresh. At `ADDON_LOADED`, `TacoTipConfig.custom_pos` is checked and `TacoTip_CustomPosEnable(false)` is called, which recreates the button and syncs its position. **No gap here.**

2. **Gap: checkbox ↔ sync race.** When the positioning checkbox toggles custom position ON, it calls `TacoTip_CustomPosEnable(false)` (create+enable but HIDE). The button exists but is hidden. The tooltip anchor system doesn't check whether the mover is visible — it checks `TacoTipConfig.custom_pos`. So the tooltip re-anchors to a HIDDEN mover button. This is intentional — the mover is the anchor point even when hidden.

3. **Gap: anchor_mouse ↔ custom_pos mutual exclusion.** When custom position is turned on, `anchor_mouse` is forced to false. When mouse anchor is turned on, `custom_pos` is set to nil and mover button is disabled. The mutual exclusion is enforced in three places (option checkbox, `_Enable`, slash handler). **No divergence possible.**

4. **Gap: `syncTooltipMoverPosition` called with stale GameTooltip.** After drag end, `syncTooltipMoverPosition(true)` is called which calls `ShowExample` — but the GameTooltip might still be mid-Hide from the drag end. The `ShowExample` calls `GameTooltip:SetUnit("player")` which goes through `onTooltipSetUnit` which reads `GameTooltip:GetUnit()`. If the tooltip is in transition (hidden or just hidden), this could be stale. **Low risk** due to safeCall wrapping.

### Pass 3 — Edge Cases & Race Conditions

#### Scenario A: Mover open with tooltip delay > 0
```
TacoTipConfig.tooltip_delay = 1.5 (seconds)
/tacotip custom  → TacoTip_CustomPosEnable(true) → TacoTipDragButton:Show()
→ OnShow fires → ShowExample()
  → GameTooltip:SetUnit("player")
    → OnTooltipSetUnit hook fires
      → delay = 1.5 > 0
      → delayedTooltipTimer = C_Timer.NewTimer(1.5, onTooltipSetUnit)
→ Ticker (1s) fires → ShowExample()
  → GameTooltip:SetUnit("player")
    → OnTooltipSetUnit hook fires
      → cancelDelayedTooltip() — cancels PREVIOUS timer
      → delay > 0 → creates NEW timer (1.5s)
```
**Verdict:** The ticker at 1s continuously cancels and reschedules the delayed tooltip. The player tooltip would never actually render until ticker stops (mover hides). The `onMoverGameTooltipShow` hook also fires `ShowExample` on any GameTooltip show event. **Moderate bug** — tooltip delay mutes the player tooltip example during mover show. The ticker makes it perpetual.

#### Scenario B: Config reset while mover is shown
```
User has mover open with saved custom position
Type /tacotip reset or click Reset Configuration in options
→ resetCfg()
  → TacoTipDragButton:_Disable() if exists
    → Hides mover, clears custom_pos, preserves anchor
  → TacoTipConfig = GetDefaults() — custom_pos wiped
  → refreshOverlayPositions
→ (later) TacoTipDragButton reference still exists but is hidden
```
**Verdict:** Clean. `_Disable` is called first, which hides the button and kills its ticker. The reference frame remains in memory but is hidden and unhooked. Next `TacoTip_CustomPosEnable(true)` would recreate the handlers (guarded by nil check — it won't recreate the frame, but the _Enable branch re-syncs).

#### Scenario C: `/reload` with custom_pos saved
```
User has custom_pos saved
Reload UI → ADDON_LOADED fires → main.lua
  → TacoTipConfig (restored from SavedVariables) has custom_pos
  → TacoTip_CustomPosEnable(false)
    → TacoTipDragButton = nil (new session)
    → Creates TacoTipDragButton
    → Calls _Enable()
      → custom_pos exists from SV
      → syncTooltipMoverPosition(false)
      → TacoTipDragButton:Hide() (show=false param)
  → Later mouseover → GameTooltip_SetDefaultAnchor fires
    → hooksecurefunc reads TacoTipConfig.custom_pos
    → GameTooltip:SetOwner(TacoTipDragButton, ...)
    → Tooltip anchors to hidden mover button at saved position
```
**Verdict:** Works correctly. The hidden mover frame serves as the anchor point. **Structural property:** the mover button must exist (as an invisible anchor) for custom position to function.

#### Scenario D: Double-click "Mover" button
```
User clicks Mover → TacoTip_CustomPosEnable(true)
  → Guard: if TacoTipDragButton already exists, skips creation
  → _Enable() runs again
    → custom_pos exists → syncTooltipMoverPosition(false)
  → TacoTipDragButton:Show() (show param = true)
  → OnShow fires again for an already-visible button
    → Does OnShow re-fire? No — button is shown, OnShow only fires on transition
  → No issue
```
**Verdict:** Idempotent. No double-registration of script handlers (guarded by nil check).

#### Stale closures found

1. **`onMoverTicker` closure in `onDragButtonShow` (line 1702):** Captures `TacoTipDragButton` from the parent scope. Safe — the button outlives the ticker.

2. **`onMoverGameTooltipShow` closure (line 1708):** Hooks GameTooltip OnShow via `Detours:ScriptHook`. Captures `TacoTipDragButton`. NOTE: If `TacoTipDragButton:_Disable` is called without via the OnHide unhook path (it does call `onDragButtonHide` → `Detours:ScriptUnhook`), the hook is cleaned.

3. **Deferred CAfter(0.05) in `onTooltipSetUnit`:** The deferred re-border-apply in `ApplyTooltipAppearance` at line 480 captures `tooltip` by reference. If the tooltip is hidden and re-shown in < 50ms, the deferred callback could fire on a stale tooltip that's now showing different content. This is guarded by `if (not tooltip or not tooltip:IsShown()) then return end`.

### Mandatory Adversarial Pass

**Disproving my own claims:**

1. **Claim from Pass 1:** "Script handlers are registered once at creation, guarded by nil check." — ATTACK: `OnShow` script is registered *inside* the `if (not TacoTipDragButton)` guard. But `OnShow`'s closure `onDragButtonShow` calls `Detours:ScriptHook(TT, GameTooltip, "OnShow", ...)`. The `Detours:ScriptHook` documentation should be checked — does it stack multiple hooks? If yes, every call to `TacoTip_CustomPosEnable` with a new GameTooltip OnShow event would add another hook. **RETRACTED — partially.** `Detours:ScriptHook` replaces the previous hook per `TT` key. The guard is sufficient.

2. **Claim from Pass 3 (Scenario A):** "Tooltip delay permanently mutes player tooltip during mover show." — ATTACK: Is this actually true? The ticker calls `ShowExample` which calls `GameTooltip:SetUnit("player")`. The delayed tooltip handler (`onTooltipSetUnit`) runs the OLD unit handler. But `cancelDelayedTooltip` is called at the TOP of `OnTooltipSetUnit` hook (line 1062). So the pending timer IS cancelled every time SetUnit fires. Then a new timer is created. So yes — the delay perpetually kicks the can. **Claim stands, but the impact is cosmetic** (the mover example tooltip shows player data, which is correct behavior for the mover — the delay just defers it by the configured ms).

3. **Claim from Pass 3 (Scenario C):** "Mover button must exist as invisible anchor." — This is a structural property, not a bug. It's correct design. **No overclaim.**

4. **What I underclaimed:** Pass 2 identified a gap in `syncTooltipMoverPosition(true)` calling `ShowExample` while GameTooltip might be mid-transition. But there's a deeper issue: `ShowExample` calls `GameTooltip:SetUnit("player")` which, through `onTooltipSetUnit` at line 1061, calls `TT:ApplyTooltipAppearance(tooltip, tooltipUnit)` at line 1058. `ApplyTooltipAppearance` at line 480 installs a `CAfter(0.05, ...)` deferred border refresh. If `_Enable` is called (mover re-shown) or `_Disable` is called (mover hidden) within that 50ms window, the deferred callback fires on a tooltip state that doesn't match the current world state. **ADDED — low-severity race.**

5. **What I underclaimed — the 3D PlayerModel during mover:** `ShowExample` calls `GameTooltip:SetUnit("player")`. The onTooltipSetUnit handler calls `ApplyTooltipAppearance` which calls `ensureTooltipPortrait`. At line 513: `if (is3D and portrait.SetUnit and UnitIsPlayer(unit)) then pcall(portrait.SetUnit, portrait, unit)`. During mover show, the 3D player model portrait is loaded and rendered for the player's character. This is correct — but the `CAfter(0.05, ...)` deferred border-apply at line 480 references `tooltip` which could be nil/nullified by the ticker's next `GameTooltip:SetUnit("player")` call. The guard at line 482 (`not tooltip or not tooltip:IsShown()`) handles this. **No addition needed.**

**Overclaim retractions:**
- Scenario A: While technically the delay perpetually kicks, the practical effect is negligible because the mover example tooltip always shows player data regardless of delay. **Retracting the "moderate bug" classification** — downgrade to "cosmetic quirk."
- `onMoverTicker` closure: I claimed it captures `TacoTipDragButton` from parent scope — this is trivially correct and not a problem. **Removing from findings.**
